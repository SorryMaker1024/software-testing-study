# 禅道速成（Bug管理工具）

> 禅道 = 国产项目管理工具，测试岗主要用它的 Bug 管理功能。
> 面试能说"我会用禅道提Bug、跟踪状态、关Bug"即可。

---

## 一、Bug 生命周期（面试必问）

```
                   ┌── 已拒绝 ──┐
                   │            ↓
新建(New) → 确认(Active) → 修复中 → 已解决(Resolved) → 关闭(Closed)
  ↑                              ↓
  └──────── 重新打开(Reopen) ────┘
```

| 状态 | 谁操作 | 含义 |
|------|--------|------|
| 新建 (New) | 测试 | 刚提的 Bug |
| 确认 (Active) | 开发/主管 | 确认这是 Bug，开始处理 |
| 已解决 (Resolved) | 开发 | 改完了，等测试验证 |
| 关闭 (Closed) | 测试 | 验证通过，关单 |
| 重新打开 (Reopen) | 测试 | 验证没通过，打回去重新改 |
| 已拒绝 | 开发 | 开发认为不是 Bug 或不用改 |

---

## 二、面试怎么说

> "Bug 的生命周期是：测试发现 Bug → 新建 → 开发确认并修复 →
> 测试验证 → 关闭。如果验证不通过就重新打开。
> 我以前用禅道管理 Bug，提 Bug 的时候会写清楚标题、严重程度、
> 复现步骤、预期结果、实际结果、截图和环境信息。"

---

## 三、提 Bug 的要素（合格 Bug 报告必备）

| 要素 | 说明 | 示例 |
|------|------|------|
| 标题 | 简明扼要，说清楚什么模块什么问题 | 【登录页】输入正确密码后点击登录无响应 |
| 严重程度 | 致命/严重/一般/轻微/建议 | 致命=系统崩溃、严重=核心功能不能用 |
| 优先级 | 紧急/高/中/低 | 能不能延期修 |
| 复现步骤 | 数字编号，一步步写出怎么重现 | 像写操作手册 |
| 预期结果 | 应该是什么样的 | 登录成功，跳转首页 |
| 实际结果 | 实际发生了什么 | 停留在登录页，无任何提示 |
| 截图/录屏 | 附上证据 | 截图红框标注异常位置 |
| 环境 | 操作系统、浏览器版本、测试环境 | Win11 + Edge 131 + 测试环境 |

---

## 四、Bug 优先级 vs 严重程度（面试高频）

| | 严重程度 (Severity) | 优先级 (Priority) |
|------|------|------|
| 定义 | 技术角度，Bug 多严重 | 业务角度，什么时候修 |
| 举例 | 严重=登录不了，轻微=颜色偏了 | 紧急=马上修，低=下个版本修 |
| 常见误区 | 严重的 Bug 不一定优先级高 | 比如"必须用管理员密码才能触发的崩溃"，很少发生，优先级可能不高 |

---

## 五、动手实操（现在做）

### 任务：在禅道里提一个模拟 Bug

1. 登录禅道 → 进入一个项目（没有就新建：项目 → 添加项目）
2. 点 **"测试"** 标签 → **"提交 Bug"**
3. 填以下内容：

```
标题：【登录模块】登录成功后 10 秒不操作会被异常踢出
严重程度：严重（3 级）
优先级：高
复现步骤：
    1. 打开登录页
    2. 输入正确用户名和密码
    3. 点击登录，成功跳转到首页
    4. 不做任何操作，等待 10 秒
实际结果：10 秒后被强制跳回登录页，提示"登录已过期"
预期结果：10 秒不应过期，一般 Token 有效期至少 30 分钟以上
类型：代码错误
```

4. 提交后 → 截图保存，面试可以放简历做证据
