# Bug 管理笔记

> 面试必问 + 工作中每天用

---

## 一、Bug 生命周期

一条 Bug 从被发现到被关闭，走完一个完整的流程：

```
新建 → 确认/激活 → 指派给开发 → 开发修复 → 测试验证 → 关闭
                                    ↘ 拒绝/延期 → 需要产品决策
```

| 状态 | 谁操作 | 含义 |
|------|--------|------|
| **新建** | 测试 | 测试发现 Bug，刚提交，还没人看过 |
| **确认/激活** | 测试主管/测试 | 确认这真的是个 Bug，不是测试误操作 |
| **已指派** | 测试主管 | 指派给对应的开发人员 |
| **修复中** | 开发 | 开发正在改代码 |
| **已修复** | 开发 | 开发改完了，等测试来验证 |
| **已验证** | 测试 | 测试验证通过，Bug 确实修好了 |
| **关闭** | 测试 | 确认没问题，关闭这条 Bug |
| **重新打开** | 测试 | 验证发现没修好或者又复现了，重新激活 |

**还有两个特殊状态：**

| 状态 | 含义 |
|------|------|
| **拒绝** | 开发认为这不是 Bug（比如设计如此），退回来 |
| **延期** | 承认是 Bug，但优先级低，这个版本不修，下个版本再说 |

**面试话术**：
> "Bug 的生命周期就是从测试发现 Bug 开始，提交后由测试主管确认并指派给开发，开发修复后交给测试验证，验证通过则关闭。如果开发认为是设计如此可以拒绝，如果优先级低可以延期到下个版本。"

## 二、优先级 vs 严重程度（面试高频题）

很多人搞混这两个概念：

| 维度 | 优先级 Priority | 严重程度 Severity |
|------|----------------|-------------------|
| **回答什么** | 应该**多快**修？ | 影响**多大**？ |
| **谁定** | 产品经理/项目经理 | 测试人员 |
| **会不会变** | 随版本进度变化 | 一般不轻易改 |

### 优先级分 4 档

| 优先级 | 含义 | 举例 |
|--------|------|------|
| P0-紧急 | 阻塞测试/上线，立刻修 | 登录功能崩了，所有人登不进去 |
| P1-高 | 严重影响用户使用，本版本必须修 | 支付金额算错 |
| P2-中 | 影响体验但不致命，通常要修 | 某个页面样式错乱 |
| P3-低 | 小瑕疵，有时间就修 | 文案多了一个句号 |

### 严重程度分 4 档

| 严重程度 | 含义 | 举例 |
|----------|------|------|
| S0-致命 | 系统崩溃、数据丢失、安全问题 | 删除用户数据、SQL注入 |
| S1-严重 | 核心功能不可用，但没有数据损失 | 下单按钮点不了 |
| S2-一般 | 功能可用但有明显问题 | 搜索结果不准确 |
| S3-轻微 | 不影响使用的小问题 | 提示文字不友好、UI小瑕疵 |

### 常见的"优先级≠严重程度"的例子

| 场景 | 严重程度 | 优先级 | 为什么 |
|------|----------|--------|--------|
| 首页 Logo 拼错了 | S3 轻微 | P0 紧急 | 影响公司形象，必须立刻改 |
| 某深层隐藏页面的404 | S2 一般 | P3 低 | 用户几乎看不到，下版本再修 |
| 支付金额多了一分钱 | S0 致命 | P0 紧急 | 涉及钱，严重程度和优先级双高 |

**面试话术**：
> "严重程度是 Bug 本身造成的影响有多大，由测试人员判定；优先级是修 Bug 的紧急程度，由产品经理和项目经理决定。两者通常正相关，但也有特例——比如首页 Logo 拼错，严重程度很低但优先级很高，因为这影响公司形象。"

## 三、Bug 报告的要素

一条合格的 Bug 报告必须包含以下内容：

### 必须有的字段

| 字段 | 说明 | 好的例子 | 坏的例子 |
|------|------|----------|----------|
| **标题** | 简明描述问题 | `【登录页】输入正确账号密码后点击登录无响应` | `登录不好使` |
| **严重程度** | S0/S1/S2/S3 | S1 | - |
| **优先级** | P0/P1/P2/P3 | P1 | - |
| **所属模块** | 哪个功能模块 | 登录模块 | - |
| **发现版本** | 测试的版本号 | v1.2.0 | - |
| **复现步骤** | 一步一步写清楚 | 1.打开登录页 2.输入xxx 3.点击登录 | `登录报错` |
| **预期结果** | 应该是什么样 | 跳转到首页，显示用户名 | `正常` |
| **实际结果** | 实际是什么样 | 页面无任何反应，控制台报错 500 | `坏了` |
| **环境信息** | 浏览器/系统/分辨率等 | Chrome 120 / Windows 11 | - |
| **截图/录屏** | 有图最直观 | 附 Bug 截图 | - |

### 写 Bug 标题的技巧

```
格式：【模块名】条件 → 现象

✅ 【登录页】输入正确账号密码，点击登录按钮无响应
✅ 【购物车】选择3件商品，结算金额计算错误，显示为负数
✅ 【搜索】输入超长字符串（>1000字符），页面崩溃白屏

❌ 登录有问题
❌ bug
❌ 修一下登录
```

### 写复现步骤的技巧

- 像写菜谱一样，别人照着做就能复现
- 用 1、2、3、4 编号
- 如果某些步骤有前提条件，写清楚

```
【好的复现步骤】
前置条件：已有账号 test001/123456
1. 打开 http://xxx.com/login
2. 输入用户名 test001，密码 123456
3. 点击"登录"按钮
4. 观察页面反应

预期结果：登录成功，跳转到首页 http://xxx.com/home
实际结果：页面无响应，按钮变灰但未跳转
复现率：5/5 次（100%）
```

## 四、Bug 管理工具

### 主流工具

| 工具 | 特点 |
|------|------|
| **禅道** | 国产免费，中小公司用得最多 |
| **Jira** | 国际主流，大厂标配，功能最强 |
| **TAPD** | 腾讯出品，敏捷项目管理 |
| **飞书多维表格** | 小团队有时直接用表格管理 |

### 禅道基本使用

禅道网址：https://www.zentao.net

核心模块：
- **测试 → Bug**：提单、跟踪、验证
- **测试 → 用例**：管理测试用例
- **测试 → 测试单**：一个版本关联哪些用例和 Bug

提 Bug 流程：
1. 进入"测试 → Bug → 提Bug"
2. 选择所属产品、模块、版本
3. 填写标题、复现步骤、预期结果、实际结果
4. 选择严重程度、优先级
5. 上传截图
6. 提交

## 面试Q&A

**Q1：Bug 生命周期是什么？**
> Bug从测试发现提交，经过确认指派给开发修复，修复后测试验证，验证通过关闭。中间还有拒绝和延期两种特殊情况。

**Q2：优先级和严重程度有什么区别？**
> 严重程度衡量 Bug 造成的影响有多大（BUG本身的问题），优先级衡量修 Bug 的紧急程度（商业决策）。测试定严重程度，产品定优先级。典型例子：首页 Logo 拼错——严重程度S3轻微，但优先级P0紧急因为影响公司形象。

**Q3：Bug 报告的要素有哪些？**
> 标题、严重程度、优先级、所属模块、版本、复现步骤、预期结果、实际结果、环境信息、截图。核心是复现步骤要写清楚，让开发看了就能复现。

**Q4：开发说这不是 Bug，你怎么处理？**
> 先确认是不是自己对需求理解有误，如果确认是 Bug 但开发不认，拉产品经理确认需求文档。如果影响用户但不在当前需求范围，提为优化建议。总之以需求文档为准，拿数据说话，不跟开发吵架。
