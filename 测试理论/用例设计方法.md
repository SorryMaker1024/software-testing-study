# 测试用例设计方法

> 学习时间：Day 3 | 核心方法：等价类 → 边界值 → 判定表 → 场景法 → 错误推测

---

## 前置概念：什么是好的测试用例？

写用例之前先搞清楚什么叫"好"：

- **能发现 Bug**（最核心的目的，不是凑数量）
- **其他人能看懂、能复现**（换个人照着步骤做，结果一样）
- **步骤清晰**，不要写"测试登录"这种模糊描述

面试官判断你水平的方式：不看你会背多少定义，看你能不能**对一个具体功能写出用例**。

## 一、等价类划分（Equivalence Partitioning）

### 一句话

把海量的输入数据按"等价"分组，每组只测一个代表值。因为你不可能把所有值都测一遍。

### 核心概念

| 类型 | 定义 | 例子 |
|------|------|------|
| 有效等价类 | 合法的、程序该接收的输入 | 用户名 6-20 位 → 8位字母 |
| 无效等价类 | 非法的、程序该拒绝的输入 | 用户名 6-20 位 → 3位字母、空、超长 |

**关键原则**：一个有效等价类 + 多个无效等价类（因为出错方式各有不同）

### 举例：年龄输入框（范围 18-60）

| 等价类 | 类型 | 代表值 |
|--------|------|--------|
| ≥18 且 ≤60 | 有效 | 30 |
| <18 | 无效 | 10 |
| >60 | 无效 | 80 |
| 非数字 | 无效 | "abc" |
| 空 | 无效 | (空) |
| 小数 | 无效 | 25.5 |

> ️ **注意**：<18 和 >60 是两个不同的无效等价类，要分开测。因为它们触发的错误逻辑可能不同（一个提示"未满18岁"，一个提示"超出年龄范围"）。

### 面试这样说

> "等价类就是把输入按是否合法分组，有效等价类选一个代表，无效等价类每种错误情况各选一个代表。比如年龄限制 18-60，有效等价类我测 30，无效等价类我分别测 10、80、字母、小数、空值——每一种都可能触发不同的错误处理逻辑。"

## 二、边界值分析（Boundary Value Analysis）

### 一句话

Bug 最爱藏在边界上。边界值就是测边界上的点。

### 找边界值的口诀

对于取值范围 [a, b]（闭区间），测这 5 个点：

- **min-1**（刚好小于最小值）
- **min**（最小值本身）
- **min+1**（刚好大于最小值）
- **max-1**（刚好小于最大值）
- **max**（最大值本身）
- **max+1**（刚好大于最大值）

简化成 3 个：**min、max、min-1、max+1**，最小值本身和最大值本身也容易出问题。

### 举例：密码长度 6-20 位

| 边界点 | 值 | 预期 |
|--------|-----|------|
| 下边界 -1 | 5位密码 | 拒绝，提示"至少6位" |
| 下边界 | 6位密码 | 通过 |
| 下边界 +1 | 7位密码 | 通过 |
| 上边界 -1 | 19位密码 | 通过 |
| 上边界 | 20位密码 | 通过 |
| 上边界 +1 | 21位密码 | 拒绝，提示"最多20位" |

### 边界值不止数字

长度限制、数量限制、时间范围、文件大小……只要有"区间"，就有边界。

| 场景 | 边界怎么找 |
|------|-----------|
| 购物车最多加 99 件 | 测 0、1、98、99、100 |
| 上传图片 ≤5MB | 测 4.99MB、5MB、5.01MB |
| 搜索框最多输入 50 字符 | 测 49、50、51 |
| 分页每页 20 条 | 总数 19 条（1页）、20 条（1页满）、21 条（跨页） |

### 等价类和边界值的关系

**先划分等价类，再在等价类的边界上取值。** 两者配合，不是二选一。

### 面试这样说

> "边界值就是测边界上的点。比如密码 6-20 位，我测 5、6、20、21。实际工作中边界值出 Bug 概率最高——开发经常写成 `< 6` 而不是 `<= 6`，或者 `< 20` 而不是 `<= 20`。等价类帮我确定测哪些类型，边界值帮我确定在每个类型里具体测什么值。"

## 三、判定表 / 因果图（Decision Table / Cause-Effect）

### 一句话

当结果由**多个条件的组合**决定时，画张表把所有组合列出来，确保没有遗漏。

### 什么时候用

- 一个功能的结果受多个输入条件影响
- 条件之间有逻辑组合（与、或、非）
- 需求文档里一大堆"如果……同时……则……否则……"

### 举例：超市会员折扣规则

> **规则描述**：
> - 会员 + 消费 ≥ 500 元 → 打 8 折
> - 会员 + 消费 < 500 元 → 打 9 折
> - 非会员 + 消费 ≥ 500 元 → 打 9.5 折
> - 非会员 + 消费 < 500 元 → 不打折

**第一步：找条件桩和动作桩**

- 条件：是否会员（是/否）、消费金额（≥500 / <500）
- 动作：打折方式

**第二步：画判定表**

| 条件\规则 | 1 | 2 | 3 | 4 |
|-----------|---|---|---|---|
| 是否会员 | 是 | 是 | 否 | 否 |
| 消费 ≥500 | 是 | 否 | 是 | 否 |
| **动作** |   |   |   |   |
| 打折 | 8折 | 9折 | 9.5折 | 无折扣 |

2 个条件 × 各 2 种取值 = 4 条规则，不会多也不会少。

**第三步：合并化简（如果两条规则结果相同、条件只差一个，可以合并）**

这 4 条规则动作都不同，不能合并。如果规则 3 和规则 4 都是"无折扣"，就可以把"消费金额"那行合并掉，变成"非会员 → 不管消费多少都不打折"。

### 关于因果图

因果图就是判定表的"可视化版"——用图画出条件（因）和结果（果）之间的与/或/非关系。实际工作中**大部分人直接用判定表**，因为画图慢且复杂。面试时提到因果图就行，重点会画判定表。

### 面试这样说

> "当功能受多个条件组合影响时，用判定表可以保证不遗漏——把条件全部列出来，每个条件取值的组合都对应一条规则。比如会员折扣有 2 个条件，各 2 种取值，就是 4 条规则，每条规则的输出都明确。因果图本质是判定表的图形化，实际工作中直接用判定表更高效。"

## 四、场景法（Scenario Method）

### 一句话

模拟用户使用软件的真实流程来设计用例——用户在实际中怎么操作，你就怎么测。

### 核心概念

| 概念 | 定义 | 例子（ATM取款） |
|------|------|----------------|
| **基本流** | 用户按预期操作，一路顺利到底 | 插卡 → 输密码 → 选取款 → 输金额 → 拿钱 → 退卡 |
| **备选流** | 中间某个环节出了岔子，但能回到基本流继续 | 密码输错 1 次 → 重新输入 → 正确 → 继续 |
| **异常流** | 出岔子后无法继续，流程终止 | 密码输错 3 次 → 吞卡 → 结束 |

### 举例：电商下单流程

**基本流**：打开商品页 → 加入购物车 → 去结算 → 填地址 → 选支付方式 → 确认支付 → 支付成功 → 看到订单

**备选流**：

| 从哪步分离 | 做什么 | 预期 |
|-----------|--------|------|
| 加入购物车 | 库存不足，提示仅剩 1 件 | 用户决定要不要加 |
| 去结算 | 购物车空了（商品过期下架） | 提示"商品已失效" |
| 填地址 | 新增收货地址 | 地址保存成功，回到结算页 |
| 选支付 | 余额不足 | 提示切换支付方式 |
| 支付中 | 支付超时 | 订单状态为"待支付"，可重新发起 |

**异常流**：支付时网络断开、银行系统故障、支付过程中 App 被杀死

### 面试这样说

> "场景法就是模拟用户真实使用流程来写用例。核心是区分基本流、备选流和异常流——基本流是一切顺利的主路径，备选流是中间出小岔子但能恢复，异常流是出大问题流程终止。面试的时候如果让你测一个功能，先描述基本流，再补充几个关键备选流，虽然简单但体现你有结构化思维。"

## 五、错误推测法（Error Guessing）

### 一句话

靠经验和直觉猜哪里容易出 Bug，然后针对性地测。

### 常见容易出 Bug 的地方

| 经验 | 具体事例 |
|------|---------|
| 程序员容易写错等于号 | `>=` 写成 `>`，比如"满 100 减 10"恰好 100 元不触发 |
| 空值没处理 | 搜索框什么都不输入点搜索 → 崩溃或全表查询 |
| 空格没 trim | 输入" 13800001111"（前面有空格）→ 提示格式错误 |
| 特殊字符 | 输入 `<script>alert(1)</script>` → XSS |
| 并发操作 | 双指同时点"提交订单" → 扣两次钱 |
| 回退/刷新 | 支付成功后点浏览器后退 → 重复下单 |
| 零值和负数 | 数量输入 -1 或 0 → 价格变负数 |
| 极大值 | 一个订单里放 99999 件商品 → 总价溢出 |

> 这个方法**依赖经验**，测得多了自然积累。刚开始学习时不用焦虑，把上面列举的这些记住，面试够用了。

## 六、面试Q&A

### Q1：等价类和边界值有什么区别？

> "等价类解决'测哪些类型'的问题——把输入分成有效和无效两类；边界值解决'每个类型测什么具体值'的问题——在等价类的边界上取值。两者配合使用：先划分等价类，再对每个等价类选边界值。实际中边界值发现 Bug 的概率最高。"

### Q2：什么情况下用判定表？

> "当一个功能的输出由多个条件的组合决定时。比如会员折扣同时看'是否会员'和'消费金额'两个条件。条件越多，人越容易漏，判定表强制你把所有组合列出来。N 个条件各 2 种取值，就是 2^N 条规则，一条都不会少。"

### Q3：基本流和备选流怎么区分？

> "基本流是一路顺利到底的理想路径；备选流是中间某个环节偏离了，但能纠正回来继续走下去；异常流是流程被迫终止。比如 ATM 取款，输错一次密码是备选流，输错三次吞卡是异常流。"

### Q4：给你一个登录页，你怎么设计测试用例？

> "分 5 个维度：第一，界面 UI——元素是否完整、对齐、按钮是否可点击；第二，功能正常流——正确账号密码能否登录、登录后跳转是否正确；第三，功能异常流——错误密码、空账号、空密码、不存在的账号、大小写、前后空格；第四，安全——SQL 注入、XSS、密码是否明文传输、连续错误是否锁定；第五，兼容——不同浏览器、不同分辨率。"

### Q5：你怎么判断"测够了"？

> "测够了不看用例数量，看是不是覆盖了所有等价类 + 边界值 + 关键场景流。如果每个等价类都至少有一个用例、边界值覆盖了上下边界的邻点、基本流和主要备选流都跑了，我就认为当前阶段的测试是充分的。"
