---

## 一、什么是软件测试

**定义**：在规定的条件下对程序进行操作，以发现错误、衡量软件质量、评估是否满足设计需求。

**目的**：
- 发现 Bug（不是证明软件没问题，而是尽可能多地发现问题）
- 验证软件是否满足需求
- 为产品发布提供质量信息

**核心认知**：测试不能证明软件没有 Bug，只能证明存在 Bug。测试是"找茬"，不是"证明没问题"。

---

## 二、测试流程（从需求到上线）

```
测试负责人依据迭代需求任务列表，编写测试计划：包括测试范围，时间及任务安排等
测试人员依据需求文档编写测试用例并评审
开发提测版本后，依据用例进行测试，提bug、跟踪bug、测试多轮，直至被测软件达到测试需求要求，测试结束
需求分析 → 测试计划 → 测试设计 → 测试执行 → 缺陷管理 → 测试报告
```

| 阶段 | 做什么 | 产出物 |
|------|--------|--------|
| 需求分析 | 读需求文档，理解功能，找出需求中的模糊点和矛盾点 | 需求问题清单 |
| 测试计划 | 确定测试范围、人员、时间、资源、风险 | 测试计划文档 |
| 测试设计 | 写测试用例，准备测试数据 | 测试用例文档 |
| 测试执行 | 按用例执行测试，发现 Bug 提交到禅道/Jira | Bug 清单 |
| 缺陷管理 | 跟踪 Bug 修复进度，回归验证 | 更新的 Bug 状态 |
| 测试报告 | 总结测试情况：测了什么、发现多少 Bug、还有多少没修、能不能上线 | 测试报告 |

**面试要能说**：从需求评审开始，测试就介入了。先分析需求，写测试计划和用例。开发提测后执行用例，发现 Bug 提交到禅道。开发修复后回归验证。最后出测试报告，评估是否达到上线标准。

## 三、测试分类

### 3.1 按阶段划分（软件开发生命周期）

| 阶段 | 测试对象 | 谁来做 | 核心关注点 |
|------|----------|--------|------------|
| **单元测试** | 单个函数/方法/类 | 开发人员 | 代码逻辑是否正确 |
| **集成测试** | 模块之间的接口 | 开发/测试 | 模块联调有没有问题 |
| **系统测试** | 整个系统 | 测试人员 | 功能、性能、兼容性等全面验证 |
| **验收测试** | 整个系统 | 客户/产品方 | 是否满足业务需求 |

**面试怎么说**：
- 单元测试：开发写，测代码最小单元，比如一个函数。白盒为主。
- 集成测试：测模块之间的接口调用，比如登录模块调数据库对不对。
- 系统测试：测试人员把整个系统跑起来，功能、性能、兼容性全测一遍。
- 验收测试：产品经理或客户验收，看是否满足需求。

### 3.2 按是否查看代码

| 类型 | 特点 | 适用场景 |
|------|------|----------|
| **黑盒测试** | 不看代码，只看输入输出 | 功能测试、系统测试 |
| **白盒测试** | 看代码逻辑，覆盖代码路径 | 单元测试、代码审查 |
| **灰盒测试** | 知道部分内部逻辑，但不全看 | 接口测试 |

**面试怎么说**：
- 黑盒：把软件当黑盒子，不知道里面怎么实现的，只关注"输入 → 输出"对不对。比如测登录：输入账号密码，看能不能登进去。这是测试人员最常用的方式。
- 白盒：能看见代码，根据代码逻辑设计测试。比如 if 有两个分支，两个分支都要走到。一般是开发做。
- 灰盒：介于两者之间。比如接口测试，知道接口的参数和返回值，但不知道后端具体怎么处理的。

### 3.3 按测试类型

| 类型 | 测什么 | 举例 |
|------|--------|------|
| **功能测试** | 功能是否符合需求 | 登录能不能登、订单能不能提交 |
| **性能测试** | 系统响应速度、并发能力 | 1000 人同时下单会不会卡 |
| **兼容性测试** | 不同环境能否正常运行 | 不同浏览器（Chrome/Firefox/Edge）、不同手机型号 |
| **安全测试** | 有没有安全漏洞 | SQL 注入、XSS 攻击、越权访问 |

### 3.4 其他重要概念

| 概念 | 一句话解释 | 什么时候用 |
|------|-----------|------------|
| **冒烟测试** | 上线前快速跑一遍核心功能，确认"能不能用" | 每次提测后先冒烟，冒烟不过直接打回 |
| **回归测试** | 改了代码后重新测一遍，确认旧功能没坏 | 每次 Bug 修完、版本更新后 |
| **探索性测试** | 不按用例，凭经验和直觉自由探索，找意外 Bug | 用例跑完后，随意点点看有没有异常 |

### 3.5 α测试 和 β测试

| 类型 | 环境 | 测试人员 | 特点 |
|------|------|----------|------|
| **α测试（Alpha）** | 公司内部 | 测试人员/内部员工 | 开发在场，有问题当场改 |
| **β测试（Beta）** | 用户真实环境 | 真实用户 | 开发不在场，收集用户反馈 |

**面试一句话**：Alpha 是内部验收，Beta 是外部公测。比如微信新版本，先内部测（Alpha），再推给部分用户灰度发布（Beta）。

## 四、面试高频问题速记

### Q1：测试流程是什么样的？
从需求分析开始，参与需求评审 → 写测试计划 → 设计测试用例 → 开发提测后执行用例 → 发现 Bug 提交到禅道 → 开发修完回归验证 → 出测试报告。

### Q2：黑盒和白盒有什么区别？
黑盒不看代码只看功能，是测试人员的主要工作方式；白盒看代码逻辑，一般是开发做单元测试。

### Q3：冒烟测试和回归测试的区别？
冒烟测的是"能不能用"（核心功能是否正常），回归测的是"有没有坏"（改了代码后旧功能是否受影响）。冒烟时间短（几分钟到半小时），回归时间长。

### Q4：什么时候不适合做自动化？
- 需求频繁变动的功能（维护脚本的成本太高）
- 只测一两次的功能（投入产出比低）
- 用户体验/界面美观度相关的测试（需要人眼看）
- 探索性测试（依赖人的经验和直觉）
