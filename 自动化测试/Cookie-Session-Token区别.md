# Cookie / Session / Token 认证区别

> 面试必问。一句话：Cookie 是身份证，Session 是派出所档案，Token 是通行证。

---

## 一、Cookie（浏览器端存的小纸条）

```
你登录成功 → 服务器在你浏览器里贴了个小纸条（Cookie）
下次访问 → 浏览器自动带上这个小纸条
服务器看到纸条 → 认出了你
```

**特点**：
- 存在**浏览器**里，每次请求自动带
- 大小限制 4KB
- 可以被 js 读取（有安全风险）
- 由浏览器自动管理，不用写代码

---

## 二、Session（服务器端存的档案）

```
你登录成功 → 服务器在数据库里记了一笔档案（Session）
同时往你浏览器 Cookie 里塞了档案编号（Session ID）
下次访问 → 浏览器带 Session ID → 服务器查档案 → 认出是你
```

**特点**：
- 数据存在**服务器**，浏览器只存一个 ID
- 相对安全（真实数据不在用户手里）
- 服务器要存储，用户多了撑内存
- 分布式部署要共享 Session（麻烦）

---

## 三、Token（你自己带的通行证）

```
你登录成功 → 服务器发你一个 Token（通行证）
下次访问 → 你自己在请求头里带上 Token
服务器验证 Token 真假 → 认出是你，不需要查数据库
```

**特点**：
- 存在**客户端**（localStorage / 内存），不是 Cookie
- 请求时要手动加到 Header 里：Authorization: Bearer xxx
- 服务器不用存（无状态），天生适合分布式
- 最常见的是 JWT（JSON Web Token），中间一段是加密后的用户信息

---

## 四、对比表

| | Cookie | Session | Token (JWT) |
|--------|--------|---------|-------------|
| 数据存哪 | 浏览器 | 服务器 | 客户端 |
| 自动还是手动 | 自动带 | Cookie自动带Session ID | 手动加 Header |
| 服务器压力 | 无 | 大（要存储） | 无（无状态） |
| 安全性 | 低（能被js读） | 中（只暴露ID） | 高（加密签名） |
| 移动端/APP | 不友好 | 不友好 | ✅ 天然支持 |
| 分布式/微服务 | 要共享Cookie | 要共享Session | ✅ 天然支持 |

---

## 五、面试口答（背这段）

> 区别主要有三点：
>
> 第一，**存的位置**不同。Cookie 存在浏览器，Session 存在服务器，Token 存在客户端自己保管。
>
> 第二，**验证方式**不同。Cookie 浏览器自动携带；Session 通过 Cookie 传 Session ID，服务器查数据库验证；Token 是客户端手动放到 Header 里，服务器用密钥验签，不需要查数据库。
>
> 第三，**适用场景**不同。单体应用老项目多用 Cookie+Session，前后端分离、移动端 APP、微服务架构现在基本都用 Token。
>
> 现在主流做法是登录后拿 JWT Token，存 localStorage，每次请求设 `Authorization: Bearer <token>`。
